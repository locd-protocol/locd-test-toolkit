## What Was Just Completed: locd-verification

### Implementation Summary

Implemented the complete challenge-response verification protocol (spec §7):

#### 1. Protocol Messages (`messages.rs`)
All 4 CBOR-encoded message types:
- **HelloMessage**: Claimant → Verifier
  - Identity domain
  - Device public key (32 bytes)

- **ChallengeMessage**: Verifier → Claimant
  - 32-byte random nonce
  - Unix timestamp
  - Verifier's identity domain

- **ResponseMessage**: Claimant → Verifier
  - Ed25519 signature over (nonce || timestamp || verifier_domain)
  - COSE Sign1 delegation token
  - Optional sub-delegation chain (max depth 3)

- **ResultMessage**: Verifier → Claimant
  - Verified (true/false)
  - Reason code (13 possible values)
  - Optional WireGuard public key (if verified)

#### 2. Claimant Logic (`claimant.rs`)
- Generate HELLO messages
- Create signed responses to challenges
- Validate timestamp tolerance (60 seconds)
- Handle verification results

#### 3. Verifier Logic (`verifier.rs`)
- Process HELLO and generate CHALLENGE
- Complete verification flow:
  - DNS Master Key lookup (placeholder - async)
  - Delegation signature verification
  - Expiry checking
  - Revocation checking (pluggable trait)
  - Scope validation (service/action)
  - Response signature verification
  - Timestamp tolerance checking
  - Sub-delegation chain depth validation

#### 4. Test Coverage
**23 tests** covering:
- Message encoding/decoding roundtrips
- Challenge generation with random nonces
- Timestamp validation (within/outside 60s tolerance)
- Signature verification
- Revocation checking (mock implementation)
- Chain depth validation
- End-to-end message flow
- All 13 reason codes

### Key Architecture Decisions

1. **DNS Lookup Placeholder**: The `lookup_master_key()` function is a placeholder because `locd-dns::DnsResolver::query_identity()` is async, but the verification flow is currently synchronous. This would be refactored to async in production or use a synchronous mock for testing.

2. **Pluggable Revocation**: `RevocationChecker` trait allows different revocation strategies:
   - DNS TXT records
   - HTTPS supplementary lists
   - Cache with TTL
   - Mock implementation for testing

3. **CBOR Map Handling**: Used `Vec<(Value, Value)>` pattern for CBOR maps (not BTreeMap), consistent with locd-delegation.

4. **Ownership Patterns**: Ed25519KeyPair doesn't implement Clone, so tests extract public keys before moving keypairs into Claimant/token builders.

### 3. locd-delegation (Tokens) ✅

**Purpose**: Delegation token creation, signing, validation.

**Key features**:
- CBOR encoding with integer keys (spec §6.2)
- COSE-like signature structure (simplified)
- Builder pattern for token creation
- Constraint validation (services, actions, expiry, max_uses)
- Timestamp tolerance checking

**Key exports**:
- `DelegationToken`, `DelegationTokenBuilder`
- `DelegationValidator`, `ValidationContext`

**Example usage**:
```rust
let token = DelegationToken::builder()
    .delegator(master_key.public_key().to_bytes())
    .delegate(device_key.public_key().to_bytes())
    .expires_in(86400)  // 24 hours
    .service("api.example.com")
    .action("read")
    .build()?;

let signed = token.sign(&master_key)?;
let verified = DelegationToken::verify(&signed, &master_key.public_key())?;
```

### 4. locd-dns (DNS Records) ✅

**Purpose**: DNS TXT record formatting/parsing for Loc'd protocol.

**Record types**:
- **IdentityRecord**: `_locd.<domain>` - Master Key publication
- **RevocationRecord**: `_locd-revoke.<domain>` - Revoked delegation IDs
- **RotationRecord**: `_locd-rotate.<domain>` - Key rotation proofs

**Format examples**:

```
_locd-revoke.example.com. 60 IN TXT "v=locd-revoke1; ids=uuid1,uuid2; t=1234567890"
_locd-rotate.example.com. 300 IN TXT "v=locd-rotate1; old=<key>; new=<key>; sig=<sig>"
```

**Key exports**:
- `IdentityRecord`, `RevocationRecord`, `RotationRecord`
- `DnsResolver`, `QueryOptions` (placeholder for DNSSEC)

**Note**: DNS resolver is async and returns placeholder errors. Real implementation would use `trust-dns-resolver` with DNSSEC validation.

### 5. locd-verification (Challenge-Response) ✅ NEW

**Purpose**: Challenge-response verification protocol (spec §7).

**Protocol flow**:
```
Claimant                                    Verifier
   │──── 1. HELLO (domain, pubkey) ─────────▶│
   │                                          │ 2. DNS lookup
   │◀─── 3. CHALLENGE (nonce, timestamp) ────│
   │                                          │
   │──── 5. RESPONSE (sig, delegation) ──────▶│
   │                                          │ 6. Verify all
   │◀─── 8. RESULT (verified, reason) ────────│
```

**Key exports**:
- `HelloMessage`, `ChallengeMessage`, `ResponseMessage`, `ResultMessage`
- `ReasonCode` - 13 verification failure reasons
- `Claimant`, `VerificationOutcome`
- `Verifier`, `RevocationChecker` (trait), `MockRevocationChecker`

**Example usage**:
```rust
// Claimant creates hello
let device_key = Ed25519KeyPair::generate();
let device_pubkey = device_key.public_key().to_bytes();
let claimant = Claimant::new(device_key, IdentityDomain::new("alice.com"));
let hello = claimant.create_hello()?;

// Verifier generates challenge
let verifier = Verifier::new(
    IdentityDomain::new("service.com"),
    wireguard_pubkey,
    Some(Box::new(revocation_checker)),
);
let challenge = verifier.handle_hello(&hello)?;

// Claimant responds
let response = claimant.create_response(&challenge, delegation_token, Vec::new())?;

// Verifier validates
let result = verifier.verify_response(&hello, &challenge, &response, "service.com", "read")?;
```

## Common Issues & Solutions

### Issue: CBOR map key lookups
**Problem**: `ciborium::Value::Map` is a `Vec<(Value, Value)>`, not a map.
**Solution**: Iterate through vec, match on `Value::Integer(key)`.

### Issue: Ed25519KeyPair ownership
**Problem**: `Ed25519KeyPair` doesn't implement `Clone`.
**Solution**: Extract public key bytes before moving keypair:
```rust
let device_key = Ed25519KeyPair::generate();
let device_pubkey = device_key.public_key().to_bytes();
let claimant = Claimant::new(device_key, domain); // moves device_key
// Use device_pubkey for delegation token builder
```

### Issue: IdentityDomain construction
**Problem**: `IdentityDomain(String)` is a tuple struct with private field.
**Solution**: Use `IdentityDomain::new("domain")` constructor.

### Issue: Error variants
**Problem**: `Error::InvalidFormat` doesn't exist.
**Solution**: Use `Error::Custom("message")`, `Error::Encoding()`, `Error::Decoding()`, etc.

### Issue: ValidationContext fields
**Problem**: Fields are `Option<String>` and require `use_count` field.
**Solution**:
```rust
let context = ValidationContext {
    service: Some("service.com".to_string()),
    action: Some("read".to_string()),
    use_count: None,
    current_time: timestamp,
};
```

### Issue: DelegationValidator is static
**Problem**: `DelegationValidator::new()` doesn't exist.
**Solution**: Use `DelegationValidator::validate(&token, &context)` (static method).

### Issue: Async DNS in sync context
**Problem**: `DnsResolver::query_identity()` is async but verifier is sync.
**Solution**: Currently a placeholder returning error. For production:
- Option 1: Make verifier async
- Option 2: Use blocking runtime wrapper
- Option 3: Inject synchronous DNS mock for testing
